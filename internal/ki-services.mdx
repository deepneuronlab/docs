## Übersicht

Notes Auditor nutzt drei spezialisierte KI-Services:

## Delphi — LLM-Dokumentenanalyse

**Port:** 3211 | **Python:** 3.11

Delphi nutzt Retrieval-Augmented Generation (RAG) zur Analyse von Prüfungsdokumenten und zur Beantwortung von Checklisten-Fragen.

**Pipeline:**
1. Dokument wird in Chunks aufgeteilt (chunking)
2. Chunks werden als Vektoren in Qdrant gespeichert
3. Bei einer Frage werden relevante Chunks per Vektor-Suche abgerufen
4. LLM (Azure OpenAI) generiert Antwort basierend auf den Chunks
5. LLM-basiertes Reranking für optimale Ergebnisse

**Outputs:** Antworten auf Checklisten-Fragen, Referenzen im Dokument, Konsistenz-Vorschläge

**Prompts:** Lokalisiert (de/en) in `delphi/src/delphi/prompts/`

## Spectra — Zahlenerkennung

**Port:** 5379 | **Python:** 3.12

Spectra erkennt finanzielle Zahlen in Dokumenten und extrahiert deren Attribute.

**Pipeline:**
1. Regex-basierte Zahlenerkennung im Dokument
2. KI-Attribut-Extraktion (Multiplikator, Typ, Entität, Datum)
3. FAISS-basierte Vektor-Suche für externe Nachweise
4. Embedding-Cache in Redis (14 Tage TTL)

**Outputs:** Erkannte Zahlen mit Attributen, Verknüpfungen zu Nachweisen

**Operationen:** +, −, ×, ÷, Δ% (prozentuale Änderung)

### Spectra-Endpunkte

| Methode | Pfad | Beschreibung |
|---|---|---|
| POST | `/search-numbers` | Zahlen im Dokument finden (Regex, kein KI) |
| POST | `/analyze-numbers` | Attribute für Zahlen bestimmen (via OpenAI) |
| POST | `/propose-evidences` | Nachweise zwischen Audit- und Support-Zahlen vorschlagen |
| POST | `/calculate_consistency` | Konsistenz zwischen zwei Zahlen berechnen |
| GET | `/healthcheck` | Health-Check |
| GET | `/readiness` | Readiness-Probe mit Speicher-Prüfung |

### Unterstützte Dokumentformate

Spectra unterstützt die folgenden Dokumentformate über den `document_format`-Parameter:

| Format | Wert | Hinweise |
|---|---|---|
| PDF | `pdf` | Standard-Dokumentformat |
| Excel | `excel` | `text` und `paragraphs` müssen leer sein; Tabellen müssen Zellen enthalten |
| Word | `word` | — |

### Unterstützte Sprachen (Locale)

| Locale | Wert |
|---|---|
| Deutsch | `de-DE` |
| Englisch | `en-GB` |
| Schwedisch | `sv-SE` |
| Tschechisch | `cs-CZ` |

### Zahlen-Attribute (`NumberAttributes`)

Jede erkannte Zahl kann folgende Attribute haben:

| Feld | Typ | Beschreibung |
|---|---|---|
| `multiplier` | enum | Multiplikator: `1`, `1000`, `1000000`, `1000000000`, `1000000000000`. Ungültige Werte werden auf `null` zurückgesetzt. |
| `kind` | enum | Art der Zahl: `money`, `percentage`, `count`, `unknown`, `temporal`, `document_reference` |
| `entity` | string? | Beschreibung der Bedeutung (z.B. „Anzahl Mitarbeiter") |
| `date` | string? | Bezugsdatum (z.B. `2024-12-31`) |
| `numeric_value` | decimal? | Mathematischer Wert (z.B. 10 TEUR → `10000`) |

<Note>
  Wenn `kind` den Wert `percentage` hat, wird `multiplier` automatisch auf `1` (ONE) gesetzt. Dies verhindert fehlerhafte Berechnungen bei prozentualen Werten.
</Note>

### Nachweis-Vorschlag (`/propose-evidences`)

Der Endpunkt unterstützt sowohl **externe** als auch **interne Nachweise**:

- **Externe Nachweise:** Audit-Zahlen aus einem Prüfungsdokument werden mit Zahlen aus Support-Dokumenten abgeglichen.
- **Interne Nachweise:** Wenn das Audit-Dokument auch in `support_documents` enthalten ist, werden transitive Verbindungen automatisch berechnet (z.B. wenn A↔B und B↔C, dann auch A↔C).

Jeder Nachweis hat einen `reference_state` (`proposal` oder `accepted`) und eine `consistency` (`consistent`, `inconsistent` oder `none`). Bei transitiven Verbindungen gilt die konservativste Bewertung.

### Konsistenzprüfung (`/calculate_consistency`)

Prüft ob zwei Zahlen (Audit und Support) numerisch konsistent sind, basierend auf ihren Attributen. Fehlende Attribute werden aus dem Text abgeleitet. Gibt `consistent` oder `inconsistent` zurück.

## 3. Pythia — ML-Scoring

**Port:** 5378 | **Python:** 3.10

Pythia bietet Binärklassifikation (Ja/Nein) für Checklisten-Fragen, basierend auf historischen Antwort-Daten.

**Inputs:** Frage + Dokumentkontext
**Outputs:** Wahrscheinlichkeits-Score für Ja/Nein

## 4. Mentat — Kontenzuordnung

**Port:** 3212 | **Python:** 3.11

Mentat ordnet Konten aus der Buchhaltung den Positionen in der Taxonomie zu. Nutzt einen 6-stufigen Matching-Ansatz:

<Steps>
  <Step title="Positionsname">
    Direkte Namensübereinstimmung
  </Step>
  <Step title="Mathematisches Matching">
    3 Strategien für Summenabgleich
  </Step>
  <Step title="Vektor-Suche">
    Ähnliche Konten via Qdrant
  </Step>
  <Step title="ML-Klassifikation">
    FastText-basierte Zuordnung
  </Step>
  <Step title="LLM-Hierarchie">
    Hierarchische Traversierung via Azure OpenAI
  </Step>
  <Step title="Manuell">
    Fallback für nicht automatisch zuordenbare Konten
  </Step>
</Steps>

## Konfiguration (Environment Variables)

```
ENABLE_SPECTRA_PREDICTIONS=true
ENABLE_DELPHI_PREDICTIONS=true
ENABLE_MENTAT_PREDICTIONS=true
ENABLE_STD_CALCULATIONS_PREDICTIONS=true
ENABLE_LLM_CALCULATIONS_PREDICTIONS=true
ENABLE_LLM_DIFF_CLASSIFICATIONS=true
LLM_CALCULATIONS_MAX_CONCURRENT_DOCUMENTS=1
LLM_CALCULATIONS_MAX_CONCURRENT_REQUESTS=4
```
